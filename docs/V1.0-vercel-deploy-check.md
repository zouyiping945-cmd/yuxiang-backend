# V1.0-B Vercel 部署前代码检查

## 1. 部署前检查结果

当前 Next.js 后端项目已完成 Vercel 部署前静态检查。

- `package.json` 已包含 `dev`、`build`、`start` 脚本。
- `build` 脚本为 `next build`，可供 Vercel 构建使用。
- `next.config.mjs` 当前为最小空配置，没有发现影响 Vercel 部署的自定义项。
- `app/api/plan/route.ts` 不依赖本地文件写入。
- `app/api/plan/route.ts` 不依赖 `localhost`、`127.0.0.1` 或 `172.20.10.4`。
- `lib/data/supabase-villages.ts` 只在服务端读取 Supabase，使用环境变量读取密钥。
- Supabase 读取失败、环境变量缺失或数据为空时，会 fallback 到本地 `REAL_VILLAGES`。
- `/api/plan` 返回结构保持 V0.9 兼容。

## 2. 需要配置的 Vercel 环境变量

Vercel 项目环境变量至少需要配置：

```text
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
```

如果线上要启用 DeepSeek provider，还需要配置：

```text
AI_PROVIDER=deepseek
DEEPSEEK_API_KEY=
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
```

如果继续使用 mock / seed 推荐，可以暂时保持：

```text
AI_PROVIDER=mock
```

注意：

- `SUPABASE_SERVICE_ROLE_KEY` 只能放在 Vercel 后端环境变量中。
- 不要把 `SUPABASE_SERVICE_ROLE_KEY` 写入前端项目。
- 不要使用 `NEXT_PUBLIC_SUPABASE_SERVICE_ROLE_KEY`。
- 小程序端不需要知道 Supabase key。

## 3. Vercel 部署后的 API 测试方式

部署完成后，先测试：

```text
POST https://your-vercel-domain.vercel.app/api/plan
```

请求体示例：

```json
{
  "inputText": "周末想带父母轻松走走，想吃农家菜，不想太累",
  "companions": ["带老人 / 亲子"],
  "demands": ["轻松慢游", "想吃农家菜", "不想太累"]
}
```

预期返回：

```json
{
  "ok": true,
  "data": {
    "summary": "string",
    "matchScore": 88,
    "recommended": {},
    "routeOptions": [],
    "foods": [],
    "stays": [],
    "alternatives": []
  }
}
```

验收重点：

- `ok` 为 `true`。
- `data.summary` 正常中文，无乱码。
- `data.recommended` 存在真实中文村庄。
- `data.routeOptions` 不为空。
- `data.foods` 不为空。
- `data.stays` 不为空。
- `data.alternatives` 不为空。
- Vercel 日志中没有 Supabase 读取失败 fallback 日志。

## 4. 小程序端暂时不要切换 API_BASE_URL

V1.0-B 只检查和准备后端部署。

在 Vercel API 没有通过测试前，小程序端暂时不要切换 `API_BASE_URL`。

当前小程序本地联调仍可继续使用本地后端地址。

部署验证通过后，再进入 V1.0-C，把小程序端请求地址切换到线上 HTTPS API。
