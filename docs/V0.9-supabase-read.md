# V0.9-B Supabase 优先读取说明

## 1. V0.9-B 目标

V0.9-B 的目标是将 `/api/plan` 的推荐数据源从“只依赖本地 `REAL_VILLAGES`”升级为：

1. 优先读取 Supabase 中的真实 seed 数据。
2. Supabase 未配置、查询失败、数据为空或子表不完整时，自动 fallback 到本地 `REAL_VILLAGES`。
3. 保持 `/api/plan` 返回结构不变。
4. 不改前端 UI，不改小程序端接口调用。

## 2. Supabase 优先读取策略

新增后端读取模块：

```text
lib/data/supabase-villages.ts
```

该模块会读取以下 6 张表：

- `villages`
- `village_profiles`
- `village_designations`
- `village_routes`
- `village_foods`
- `village_stays`

读取完成后，会按 `village_id` 将多表数据重新组装成后端现有推荐逻辑使用的 `RealVillageData[]`。

## 3. REAL_VILLAGES fallback 策略

以下情况会 fallback 到本地 `REAL_VILLAGES`：

- 缺少 `SUPABASE_URL`
- 缺少 `SUPABASE_SERVICE_ROLE_KEY`
- Supabase REST API 查询失败
- `villages` 返回空数组
- 某条 village 缺少必要主表字段
- 某条 village 缺少 profile
- 某条 village 缺少 routes / foods / stays / designations

fallback 时接口不会抛出 500，仍会使用本地 seed 数据完成推荐。

## 5. 环境变量要求

后端 `.env.local` 需要配置：

```env
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
```

注意：

- `SUPABASE_SERVICE_ROLE_KEY` 只在后端使用。
- 不要暴露到 `miniapp-taro-clean`。
- 不要写死到代码里。

## 6. 风险与兜底

当前仍保留本地 fallback，因此：

- Supabase 服务异常不会导致 `/api/plan` 白屏。
- Supabase 数据为空不会影响 H5 / weapp 主链路。
- 子表缺失会触发本地 fallback，避免 `routeOptions / foods / stays` 为 `undefined`。

当前仍需注意：

- Supabase 数据仍是 seed 阶段。
- 餐饮、住宿、路线尚未人工核验。
- 暂未启用 RLS。
- 暂未接管理后台。
