# V1.1 DeepSeek AI 规划智能体接入方案

## 1. 为什么 V1.1 不继续发布，而是升级 AI 智能体

V1.0 已经完成线上后端部署、小程序线上 API 切换和发布准备文档，主链路已经具备演示与体验基础。

V1.1 的重点不再是继续推进体验版发布，而是把产品从“规则推荐 + AI 视觉包装”升级为“AI 主导解释与规划”的乡村旅游智能体雏形。

## 2. 从 AI 功能升级为 AI 主导规划

当前系统仍然保留规则推荐作为安全底座：

```text
用户需求
-> Supabase / REAL_VILLAGES 候选村庄
-> 规则打分确定 recommended + alternatives
-> DeepSeek 生成推荐解释、理由标签、出行提示
```

这样可以做到：

- 推荐目的地仍然来自真实 seed 数据。
- AI 负责生成更自然、更像规划师的解释。
- 模型不能绕过数据库自行编造村庄。
- DeepSeek 失败时，现有规则推荐仍然可用。

## 3. DeepSeek 在系统中的职责

DeepSeek 只负责增强文案，不负责决定新的村庄。

DeepSeek 输入包括：

- `inputText`
- `companions`
- `demands`
- `recommended`
- `alternatives`
- `routeOptions`
- `foods`
- `stays`

DeepSeek 输出结构：

```json
{
  "summary": "80-140字推荐理由",
  "reasonTags": ["适合父母", "轻松慢游", "农家菜友好"],
  "reasonSummary": "一句话匹配解释",
  "travelTips": ["建议错峰出行", "提前确认餐饮营业时间"]
}
```

## 4. Supabase / REAL_VILLAGES 的职责

Supabase 是线上优先数据源。

`REAL_VILLAGES` 是本地 fallback 数据源。

它们共同负责：

- 提供真实村庄候选。
- 提供结构化画像字段。
- 支撑规则打分。
- 约束 DeepSeek 只能基于候选村庄生成解释。

## 5. 为什么 DeepSeek 不能直接编造村庄

乡村旅游推荐涉及真实地点、路线、餐饮、住宿和用户出行决策。

如果模型自由编造村庄或商家，会带来：

- 用户无法找到目的地。
- 路线和价格不可信。
- 小程序演示时出现虚假内容。
- 后续上线审核和用户信任风险。

因此 V1.1 明确约束：

- `recommended.id` 只能来自后端规则推荐。
- `alternatives` 只能来自 Supabase / REAL_VILLAGES。
- DeepSeek 只能生成解释与建议。
- 不确定的信息必须使用“建议提前确认”。

## 6. Fallback 策略

DeepSeek 出现以下情况时自动 fallback：

- `AI_PROVIDER !== "deepseek"`
- `DEEPSEEK_API_KEY` 缺失
- DeepSeek HTTP 请求失败
- 请求超时
- 返回内容为空
- 返回非 JSON
- JSON 字段结构不符合要求

Fallback 后：

- `/api/plan` 不会 500。
- `recommended` 仍然正常。
- `routeOptions / foods / stays / alternatives` 仍然正常。
- `providerUsed` 保持 `mock`。
- `fallbackUsed` 在 DeepSeek 请求场景下标记为 `true`。

## 7. Netlify 需要配置的环境变量

启用 DeepSeek 需要在 Netlify 后端环境变量中配置：

```text
AI_PROVIDER=deepseek
DEEPSEEK_API_KEY=
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
```

同时仍需保留 Supabase 环境变量：

```text
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
```

注意：

- `DEEPSEEK_API_KEY` 只能放在后端。
- `SUPABASE_SERVICE_ROLE_KEY` 只能放在后端。
- 不要把任何密钥放入小程序前端项目。

## 8. 本地验收方式

本地 `.env.local` 配置 DeepSeek 后：

```text
AI_PROVIDER=deepseek
DEEPSEEK_API_KEY=your_key
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
```

测试：

```text
POST http://127.0.0.1:3000/api/plan
```

预期：

- `ok: true`
- `data.providerUsed: "deepseek"`
- `data.summary` 更自然
- `data.reasonTags` 存在
- `data.reasonSummary` 存在
- `data.recommended.id` 仍来自数据库候选村庄

断开或移除 `DEEPSEEK_API_KEY` 后：

- `/api/plan` 仍返回 `ok: true`
- `providerUsed` 回到 `mock`
- `fallbackUsed` 为 `true`

## 9. 下一步 V1.2 建议

V1.2 可以继续增强：

1. AI 对话式首页。
2. 多轮追问。

4. 展示 DeepSeek 生成的 `travelTips`。
5. 增加“为什么推荐这里”的解释卡片。
