# V0.5 真实村庄详情增强与推荐解释优化

## 1. V0.5 目标

V0.5 的目标不是继续扩展村庄数量，而是提升现有 8 个郑州及近郊 seed 村庄的数据质量和推荐解释能力。

本轮重点：

- 增强 8 个 seed 村庄详情
- 优化 tags、suitableFor、matchKeywords
- 丰富 routeOptions、foods、stays 的保守文案
- 让 `/api/plan` 的 summary 更能解释“为什么推荐这里”
- 增加可选 `reasonTags` 和 `reasonSummary`
- 优化 alternatives 的推荐解释

## 2. 为什么不继续扩数量

V0.4 已经证明真实 seed 数据可以驱动 Result 页展示，但如果继续扩数量而不提升解释质量，会出现：

- 推荐理由泛泛
- 村庄之间差异不明显
- 用户难以理解为什么推荐这里
- Result 页虽然有真实村庄名，但缺少产品说服力

因此 V0.5 先把 8 个样本做扎实，用更高质量的 seed 验证推荐解释结构，再进入 V0.6 的数量扩展。

## 3. 村庄增强字段

每个村庄重点增强：

- `description`
  - 从一句话扩展为 2-3 句产品化介绍
  - 保守表达，不编造未核实事实
- `tags`
  - 控制在 4-6 个前端友好标签
  - 突出近郊、亲子、康养、传统村落、拍照、农家菜等差异
- `matchKeywords`
  - 增加用户可能输入的自然语言
  - 覆盖父母、老人、孩子、亲子、不累、轻松、农家菜、拍照、风景、传统、文化、康养、周末、短途等
- `suitableFor`
  - 明确适合老人、亲子、家庭、朋友、情侣、研学等人群
- `routeOptions`
  - 每个村庄保留自驾和公共交通两条
  - 文案更像真实出行建议
- `foods`
  - 每个村庄至少 3 条
  - 使用附近农家菜、时令家常菜、田园茶点等保守通用名称
- `stays`
  - 每个村庄至少 2 条
  - 使用乡野民宿、近郊客栈、田园小院、山居小筑等保守通用名称

## 4. 推荐解释 summary 生成逻辑

V0.5 在 `build-plan-from-mock.ts` 中新增需求命中分析。

当前会识别：

- `hitElder`：父母、老人、长辈、爸妈
- `hitKid`：亲子、孩子、小孩、带娃
- `hitFood`：农家菜、吃饭、美食、家常菜、地锅鸡
- `hitPhoto`：拍照、出片、风景、好看、打卡
- `hitCulture`：研学、文化、传统、传统村落、历史、村落
- `hitWellness`：康养、放松、休息、散心、安静
- `hitShortTrip`：周末、短途、近郊、半日、一天
- `hitEasy`：轻松、不累、不爬山、慢游、走走、散步
- `hitFriend`：朋友、情侣、同学、同事

summary 会按以下结构生成：

1. 用户提到了什么需求
2. 当前村庄为什么匹配
3. 适合怎样安排
4. 给一个简短出行建议

示例方向：

```text
你提到带父母或长辈同行、希望轻松、不太累、想吃农家菜。樱桃沟社区更适合作为近郊短途、亲子休闲目的地，匹配适合父母、轻松不累、农家菜友好。建议按半日到一日安排，重点放在短途散步、农家菜和家庭休闲。
```

## 5. reasonTags / reasonSummary 用途

V0.5 在保持 `/api/plan` 原字段兼容的基础上，新增可选解释字段：

- `reasonTags?: string[]`
- `reasonSummary?: string`

用途：

- `reasonTags` 可用于后续前端展示“推荐命中点”
- `reasonSummary` 可用于更短的推荐解释
- 当前前端不强依赖这些字段
- 不影响 Result 页现有结构

示例：

```json
{
  "reasonTags": ["适合父母", "轻松不累", "农家菜友好"],
  "reasonSummary": "匹配你提出的带父母或长辈同行、希望轻松、不太累、想吃农家菜需求，对应适合父母、轻松不累、农家菜友好等推荐理由。"
}
```

## 6. alternatives 解释优化

V0.5 中 alternatives 不再使用完全相同的泛泛文案。

每个备选村庄会根据自身标签和当前需求生成 summary，例如：

```text
如果你更看重传统村落、乡村文化，海上桥村也适合作为备选；它与文化走看需求有一定匹配，适合做周末乡村游的替代方案。
```

同时保持：

- `matchScore` 在 88-96 区间
- 每个 alternative 都包含 summary
- 不改变 alternatives 字段名

## 7. 当前仍保持兼容

`/api/plan` 仍保留原有核心字段：

- `summary`
- `matchScore`
- `recommended`
- `routeOptions`
- `foods`
- `stays`
- `alternatives`

新增字段是可选增强，不要求前端立即使用。

## 8. V0.6 建议

V0.6 可以在 V0.5 的解释能力基础上继续扩展：

1. 扩展郑州圈 20-30 个村庄
2. 为每个村庄补充更精细的真实特色标签
3. 增加地理位置、经纬度、停车难度、适老化程度字段
4. 把 `reasonTags` 做成 Result 页低调展示
5. 引入 `preferenceHash`，支持同一村庄保存多份不同方案
6. 后续再迁移到 Supabase 或正式数据库

## 9. 注意事项

- 餐饮、住宿、路线仍是 seed 阶段的保守通用文案。
- 真实商家、价格、导航路线仍需人工核验。
- 本轮不扩展新村庄数量。
- 本轮不接 Supabase。
- 本轮不改前端 UI。
