# V0.9-A Seed 数据迁移脚本说明

## 1. V0.9-A 目标

V0.9-A 只做 Supabase 建表后的 seed 数据迁移脚本和校验脚本。

本阶段目标：

- Supabase 表结构已通过 `docs/supabase-schema.sql` 建好。
- `REAL_VILLAGES` 中的 24 个郑州圈 seed 村庄可以导入 Supabase。
- 暂不修改 `/api/plan` 读取逻辑。
- 暂不改前端。
- 暂不移除本地 `REAL_VILLAGES`。

## 2. 执行前准备

### 2.1 执行建表 SQL

先在 Supabase SQL Editor 中执行：

```sql
docs/supabase-schema.sql
```

该 SQL 会创建：

- `villages`
- `village_profiles`
- `village_designations`
- `village_routes`
- `village_foods`
- `village_stays`

### 2.2 配置环境变量

确认后端项目 `.env.local` 中存在：

```env
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
```

注意：

- 迁移脚本使用 `SUPABASE_SERVICE_ROLE_KEY`。
- 不要把 service role key 写死到代码里。
- 不要把 `.env.local` 提交到公开仓库。

### 2.3 TypeScript 脚本运行器

当前项目默认没有声明 `tsx` 依赖。

可选方式：

```powershell
npm install -D tsx
```

或者临时使用：

```powershell
npx tsx scripts/seed-real-villages-to-supabase.ts
npx tsx scripts/check-supabase-seed.ts
```

## 3. 执行迁移命令

在后端项目根目录执行：

```powershell
cd E:\乡豫app\原型图
npx tsx scripts/seed-real-villages-to-supabase.ts
```

迁移脚本会导入：

1. `villages`
2. `village_profiles`
3. `village_designations`
4. `village_routes`
5. `village_foods`
6. `village_stays`

## 4. 执行校验命令

```powershell
cd E:\乡豫app\原型图
npx tsx scripts/check-supabase-seed.ts
```

校验脚本会检查：

- `villages` 数量
- `village_profiles` 数量
- `village_designations` 数量
- `village_routes` 数量
- `village_foods` 数量
- `village_stays` 数量
- 前 3 个村庄的 profile / designation / route / food / stay 关联数量

如果 `villages` 不是 24，会输出警告。

## 5. 当前只导入数据

V0.9-A 不修改：

- `/api/plan`
- 推荐算法
- 前端页面
- 小程序接口
- 本地 fallback 逻辑

当前后端仍继续使用本地 `REAL_VILLAGES`。

## 6. 重复执行策略

迁移脚本支持重复执行：

- `villages` 使用 `village_code` upsert。
- `village_profiles` 按 `village_id` 查询后更新或插入。
- `village_designations` 会先删除该村庄旧数据，再重新插入。
- `village_routes` 会先删除该村庄旧数据，再重新插入。
- `village_foods` 会先删除该村庄旧数据，再重新插入。
- `village_stays` 会先删除该村庄旧数据，再重新插入。

这样可以避免路线、美食、住宿等子表重复堆积。

## 7. 下一步 V0.9-B

V0.9-B 建议：

1. 新增 Supabase 读取服务。
2. 将 `/api/plan` 改造为 Supabase 优先。

4. 保持 `/api/plan` 返回结构不变。
5. 增加读取失败日志和数据完整性兜底。
